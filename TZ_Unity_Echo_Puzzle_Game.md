# Техническое задание  
## Проект: 3D головоломка на C# + Unity с механикой «эхо-позиций»

---

## 4.1. Контекст проекта

### Краткое описание проекта
Необходимо разработать 3D puzzle game на **C# + Unity**, в которой игрок проходит уровни, добираясь до точки выхода, открывая путь через двери, кнопки, ловушки и другие элементы окружения.

Ключевая механика игры — **система эхо-позиций**:
- игрок может сохранить свою текущую позицию в мире как «эхо»;
- эхо отображается как полупрозрачная статичная копия персонажа;
- игрок может выбрать одно из существующих эх и выполнить **обмен позициями**;
- при обмене игрок и выбранное эхо **меняются местами**, а не происходит обычная телепортация;
- каждое эхо после обмена остаётся активным в мире и может использоваться повторно;
- количество эх ограничено, поэтому механика строится на планировании и расстановке «якорей» в пространстве.

Игра проектируется как **однопользовательская 3D-головоломка от третьего лица или от первого лица**.  
Базовое допущение для реализации в рамках данного ТЗ: **однопользовательская PC-игра для Windows**, без мультиплеера.

### Базовый технологический стек
- **Unity**: 2022.3 LTS или совместимая LTS-версия
- **Язык**: C#
- **Input**: Unity Input System
- **Physics**: стандартная 3D-физика Unity
- **UI**: uGUI или UI Toolkit (предпочтительно выбрать один подход на старте и придерживаться его во всём проекте)

### Статус проекта
Задача предполагает реализацию **с нуля**, без опоры на существующую игровую кодовую базу.

### Предлагаемая структура проекта
Ниже приведена рекомендуемая структура папок, с которой должен работать разработчик:

- `Assets/Scripts/Player/` — контроллер игрока, ввод, движение, состояние
- `Assets/Scripts/Echo/` — логика создания, хранения, выбора и обмена с эхо
- `Assets/Scripts/Interaction/` — кнопки, двери, триггеры, ловушки, платформы
- `Assets/Scripts/Level/` — логика уровня, цель, рестарт, выход
- `Assets/Scripts/UI/` — HUD, индикация активного эха, кулдауны, лимиты
- `Assets/Scenes/Levels/` — игровые уровни
- `Assets/Prefabs/` — префабы игрока, эх, объектов окружения и UI

## 4.2. Текущая реализация и проблема

### Что уже существует
На момент составления ТЗ есть:
- концепция игры;
- описание ключевых механик;
- понимание жанра: 3D puzzle / logic / traversal;
- базовое представление о том, как эхо используется для прохождения уровней.

### Что отсутствует
На данный момент отсутствуют:
- игровая архитектура;
- реализация контроллера игрока;
- система эхо;
- логика выбора активного эха;
- логика обмена позициями;
- уровень с интерактивными объектами;
- UI и система обратной связи;
- тестовая сцена и критерии готовности реализации.

### Основная проблема, которую решает задача
Нужно превратить концепт в **формализованное и реализуемое техническое решение**, чтобы разработчик мог последовательно:
1. создать базовую игровую архитектуру;
2. реализовать движение игрока;
3. реализовать механики эхо;
4. реализовать набор puzzle-элементов окружения;
5. собрать рабочие уровни, в которых механика обмена позициями является центральным способом решения задач.

### Обратная совместимость
Обратная совместимость **не требуется**, так как проект создаётся с нуля.  
При этом важно закладывать архитектуру так, чтобы:
- можно было без переработки базовой системы добавлять новые типы интерактивных объектов;
- можно было настраивать лимиты, кулдауны и правила работы эхо через конфиги;
- можно было расширять игру новыми уровнями без переписывания core-логики.

## 4.3. Цель задачи

Разработать прототип/вертикальный срез 3D-головоломки на Unity, в котором игрок проходит уровень до выхода, используя не только обычное перемещение, но и ключевую механику **создания и повторного использования эхо-позиций через обмен местами**.

Тип задачи:
- **новая функциональность**
- **архитектура и прототипирование core gameplay**
- **подготовка базы для дальнейшего масштабирования проекта**

Итогом должен стать **играбельный вертикальный срез**, демонстрирующий:
- перемещение игрока;
- создание ограниченного числа эх;
- переключение между несколькими эхо;
- обмен позициями игрока и выбранного эха;
- puzzle-объекты, взаимодействие с которыми требует грамотного размещения эх;
- прохождение уровня до финальной точки выхода.

## 4.4. Требования к функционалу

### 4.4.1. Общая игровая петля
Игровой цикл должен выглядеть следующим образом:
1. Игрок появляется на уровне в стартовой точке.
2. Игрок исследует пространство и использует обычное перемещение.
3. Игрок может создавать эхо в стратегически важных местах.
4. Игрок может выбирать активное эхо.
5. Игрок может выполнить обмен позициями с активным эхо.
6. Игрок использует эхо для обхода препятствий, удержания контроля над маршрутом и решения логических задач.
7. Игрок активирует необходимые элементы уровня и открывает путь к выходу.
8. После достижения точки выхода уровень считается завершённым.

### 4.4.2. Перемещение игрока
Необходимо реализовать базовый 3D-контроллер игрока.

#### Требования:
- перемещение по осям `X/Z`;
- поворот камеры и/или персонажа;
- прыжок — опционально, но желательно предусмотреть архитектурно;
- проверка состояния `Grounded`;
- корректная работа с коллизиями и рампами/небольшими уклонами;
- возможность запрета некоторых действий в воздухе;
- поддержка рестарта уровня.

#### Допущение:
Для прототипа допустим один из двух режимов:
- **First Person Controller**, если упор на точность платформинга и навигации;
- **Third Person Controller**, если важна визуальная читаемость обмена между игроком и эхо.

Разработчик должен выбрать один режим и реализовать его единообразно по всему проекту.

### 4.4.3. Создание эхо-позиций
Игрок должен иметь возможность по нажатию кнопки создать эхо в своей текущей позиции.

#### Требования:
- эхо создаётся в позиции игрока на момент нажатия;
- эхо сохраняет:
  - позицию;
  - поворот (при необходимости);
  - ссылку/идентификатор;
  - индекс в списке эх;
  - статус активности;
  - время создания;
- эхо визуально отображается в мире как полупрозрачная статичная копия персонажа;
- эхо не должно самостоятельно двигаться;
- эхо должно быть переиспользуемым, пока не удалено;
- количество одновременно существующих эх ограничено настраиваемым лимитом.

#### Параметры:
- `maxEchoCount` — максимум активных эх;
- `echoCreateAllowedOnlyWhenGrounded` — разрешено ли создание только на земле;
- `echoHeightOffset` — служебный оффсет для корректного отображения;
- `echoLayerMask` — слой/слои для логики выделения и выбора;
- `echoLifetime` — по умолчанию бесконечное, но предусмотреть как расширяемый параметр.

#### Правила переполнения лимита
При попытке создать эхо, когда лимит достигнут, необходимо реализовать **одну фиксированную стратегию**, выбираемую на старте проекта:
1. либо запретить создание нового эхо и показать пользователю уведомление;
2. либо удалить самое старое эхо и создать новое.

Для вертикального среза рекомендуется вариант №1, так как он лучше поддерживает puzzle-дизайн и планирование.

### 4.4.4. Выбор активного эха
Если в мире существует несколько эх, игрок должен иметь возможность выбрать, с каким из них выполнить обмен.

#### Требования:
- должна быть логика хранения списка активных эх в порядке создания;
- должно существовать понятие **активного эха**;
- игрок должен уметь переключаться между эхо:
  - по циклу вперёд/назад;
  - либо через отдельный UI-список / хоткеи;
- активное эхо должно быть визуально выделено;
- если активное эхо удалено, система должна автоматически выбрать другое доступное;
- если эх нет, состояние активного эха должно быть `null`.

#### Визуальная индикация:
- выделение контуром / цветом / усиленной прозрачностью;
- отображение индекса выбранного эха в HUD;
- отображение общего количества эх.

### 4.4.5. Обмен позициями (Swap)
Игрок должен иметь возможность мгновенно обменяться позициями с выбранным эхо.

#### Ключевое правило:
Это **не телепортация**.  
После выполнения действия:
- игрок переносится в позицию выбранного эха;
- выбранное эхо переносится в предыдущую позицию игрока;
- эхо остаётся существовать и может быть использовано повторно.

#### Требования к логике обмена:
- обмен выполняется по нажатию отдельной кнопки;
- обмен возможен только при наличии активного эха;
- перед обменом должны проверяться ограничения;
- после обмена должен запускаться кулдаун;
- должна быть визуальная и звуковая обратная связь.

#### Необходимые проверки перед swap:
- существует ли активное эхо;
- не идёт ли сейчас кулдаун;
- не находится ли игрок в запрещённом состоянии;
- валидна ли позиция для размещения игрока;
- валидна ли позиция для размещения эхо после обмена.

#### Проверки валидности позиции:
- отсутствие критического пересечения со статической геометрией;
- отсутствие попадания внутрь запрещённых объёмов;
- отсутствие размещения вне навигационно-допустимой области, если используется такое ограничение;
- отсутствие блокировки потолком/стеной при переносе капсулы игрока.

#### Требования к устойчивости:
Если swap невозможен, действие должно быть:
- отменено без частичного переноса;
- сопровождаться понятной обратной связью для игрока;
- не ломать состояние списка эх и активного выбора.

### 4.4.6. Ограничения и кулдауны
Для предотвращения хаотичного использования механики должны быть ограничения.

#### Обязательные ограничения:
- максимум одновременно существующих эх;
- задержка между swap-действиями;
- опциональный запрет создания эха в воздухе.

#### Настраиваемые параметры:
- `swapCooldownSeconds`
- `maxEchoCount`
- `allowEchoCreateInAir`
- `allowSwapInAir`
- `allowSwapWhileFalling`
- `allowSwapDuringInteraction`
- `failFeedbackDuration`

#### Поведение при запрете:
Если действие запрещено условиями:
- игра не должна выполнять действие;
- игрок должен получить обратную связь через UI/SFX/VFX;
- причина отказа должна быть понятной при отладке через лог.

### 4.4.7. Puzzle-элементы окружения
Игра должна содержать набор базовых интерактивных объектов, через которые раскрывается механика эхо.

#### Обязательный минимальный набор объектов:
1. **Кнопка / напольный триггер**
   - активируется, когда на ней находится игрок;
   - в расширенной версии может активироваться и другими объектами, но для первого вертикального среза достаточно активации игроком.

2. **Дверь**
   - закрыта по умолчанию;
   - открывается при выполнении условия;
   - может быть связана с одной или несколькими кнопками/переключателями.

3. **Опасная зона / ловушка**
   - при попадании игрока вызывает смерть, рестарт с чекпоинта или возврат к началу;
   - должна использоваться как препятствие, которое можно обойти через грамотный swap.

4. **Финальная зона выхода**
   - завершает уровень, если выполнены все условия прохождения;
   - может быть неактивна до открытия двери/активации механизма.

#### Желательные дополнительные элементы:
- подвижная платформа;
- лазер/луч с периодическим включением;
- временно открывающаяся дверь;
- переключатель, меняющий состояние уровня;
- ключ/артефакт, необходимый для открытия выхода.

### 4.4.8. Условия прохождения уровня
Система уровня должна поддерживать:
- стартовую позицию игрока;
- список обязательных условий прохождения;
- финальную точку выхода;
- рестарт уровня;
- опционально — чекпоинты.

#### Минимальный сценарий прохождения:
- игрок должен активировать один или несколько объектов;
- открыть путь;
- безопасно добраться до выхода;
- пересечь триггер завершения.

### 4.4.9. UI / HUD / обратная связь
Необходимо реализовать минимально достаточный интерфейс для понимания состояния механики.

#### HUD должен отображать:
- текущее количество созданных эх / максимальный лимит;
- индекс активного эха;
- готовность swap-умения;
- состояние кулдауна;
- служебные сообщения об ошибке действия:
  - нет активного эха;
  - достигнут лимит;
  - swap недоступен;
  - нельзя создать эхо в воздухе.

#### Визуальная обратная связь:
- эффект появления эха;
- эффект выделения выбранного эха;
- эффект успешного swap;
- эффект неуспешной попытки действия.

#### Звуковая обратная связь:
- создание эха;
- выбор эха;
- успешный swap;
- ошибка/блокировка действия;
- завершение уровня.

На первом этапе допустимо использовать временные placeholder-ресурсы.

### 4.4.10. Система состояний и обработка смерти/рестарта
Необходимо реализовать базовую систему состояний игрока и уровня.

#### Игрок:
- `Idle`
- `Move`
- `Airborne`
- `Dead` / `Disabled`
- `Swapping` (кратковременное служебное состояние при необходимости)

#### Уровень:
- `Playing`
- `Completed`
- `Failed`
- `Paused` (опционально)

#### Поведение:
- при попадании в смертельную зону игрок рестартуется;
- при рестарте уровень должен быть приведён к согласованному состоянию;
- стратегия поведения эх при рестарте должна быть заранее зафиксирована.

Для вертикального среза рекомендуется:
- **при полном рестарте уровня очищать все эхо**, чтобы избежать неоднозначных состояний.

### 4.4.11. Сохранение и конфигурирование параметров
Ключевые параметры механики должны быть вынесены из жёстко закодированных значений.

#### Требования:
- использовать `ScriptableObject` или централизованный конфиг-класс;
- хранить настройки:
  - лимит эх;
  - кулдаун swap;
  - разрешённость действий в воздухе;
  - параметры VFX/UI;
  - базовые параметры интерактивных объектов;
- обеспечить возможность быстро изменять баланс без переписывания логики.

## 4.5. Ограничения и запреты

### Запрещено
- использовать мультиплеерный стек, сетевую синхронизацию и Netcode в рамках данной задачи;
- использовать DOTS/ECS как обязательную основу проекта;
- завязывать core-механику на сторонние ассеты, без которых проект не сможет работать;
- смешивать несколько систем ввода одновременно (нужно выбрать Input System как основной вариант);
- реализовывать механику swap как простую телепортацию без обмена позициями;
- жёстко вшивать параметры лимитов и кулдаунов в десятки разных классов;
- допускать ситуации, при которых после невалидного swap игрок или эхо частично застревают в геометрии.

### Обязательные требования
- проект должен работать как минимум на **Windows PC**;
- код должен быть написан на **C# под Unity LTS**;
- архитектура должна допускать расширение новыми типами puzzle-объектов;
- ключевые действия должны быть параметризованы;
- прототип должен быть пригоден для дальнейшего масштабирования.

### Что нельзя менять без отдельного согласования
Если в будущем проект будет передан нескольким разработчикам, необходимо считать защищёнными следующие области:
- общие конфиги Input;
- базовую структуру core-сцен;
- публичные интерфейсы систем взаимодействия после их утверждения;
- соглашения по слоям (`Layers`) и тегам (`Tags`) после утверждения.

## 4.6. Требования к архитектуре и коду

### Архитектурный подход
Рекомендуемый подход: модульная архитектура с разделением по зонам ответственности.

### Обязательные принципы
- Single Responsibility для ключевых классов;
- минимум жёстких связей между системами;
- взаимодействие через ссылки, интерфейсы, события или централизованные сервисы;

## 4.8. Тестирование и критерии приёмки

### Виды тестирования
Обязательный минимум:
- ручное тестирование;
- базовые Play Mode тесты для core-механик;
- по возможности Edit Mode тесты для чистой логики менеджеров.

### Что должно быть протестировано

#### Тесты по механике эхо
1. Создание одного эхо.
2. Создание нескольких эх в рамках лимита.
3. Попытка создать эхо сверх лимита.
4. Выбор активного эха по циклу.
5. Удаление/сброс активного эха.
6. Обмен с одним эхо.
7. Повторный обмен с тем же эхо.
8. Обмен при наличии нескольких эх.
9. Запрет swap при отсутствии активного эха.
10. Запрет swap во время кулдауна.
11. Запрет создания эха в воздухе, если флаг отключён.
12. Корректность переноса позиции эхо на старое место игрока.

#### Тесты по окружению
1. Кнопка активирует дверь.
2. Дверь меняет состояние корректно.
3. Смертельная зона вызывает рестарт.
4. Выход завершает уровень только при выполненных условиях.
5. После рестарта уровень приходит в согласованное состояние.

#### UI-проверки
1. HUD показывает верное число эх.
2. HUD обновляет активное эхо.
3. HUD отображает кулдаун.
4. HUD показывает сообщение об ошибке при невалидном действии.

### Критерии приёмки
Результат считается готовым, если выполняются следующие условия:

1. Игрок может свободно перемещаться по уровню.
2. Игрок может создавать эхо в допустимых условиях.
3. В мире одновременно существует ограниченное число эх.
4. Игрок может переключаться между несколькими эхо.
5. Активное эхо визуально выделяется.
6. Swap реализован как обмен позициями, а не телепортация.
7. После swap эхо остаётся в мире на прежней позиции игрока.
8. Повторное использование эхо работает стабильно.
9. Невалидные swap-действия корректно блокируются.
10. Есть рабочий кулдаун между swap-действиями.
11. Собран как минимум один полностью проходимый puzzle-уровень.
12. Уровень нельзя пройти, полностью игнорируя механику эхо.
13. UI даёт игроку понятную информацию о состоянии механики.
14. При смерти/рестарте система возвращается в корректное состояние.
15. Проект запускается без критических ошибок в Unity и пригоден для дальнейшей доработки.

### Ручной сценарий приёмки
1. Запустить сцену `Level_01`.
2. Проверить перемещение игрока.
3. Создать первое эхо.
4. Создать второе эхо.
5. Переключиться между ними и проверить индикацию выбора.
6. Выполнить swap с первым эхо.
7. Убедиться, что эхо переместилось на старую позицию игрока.
8. Выполнить повторный swap после окончания кулдауна.
9. Проверить, что при попытке создать лишнее эхо система корректно реагирует.
10. Проверить, что без использования механики уровень не проходится.
11. Пройти уровень с использованием кнопок/дверей/опасных зон.
12. Достичь выхода и завершить уровень.

## 4.9. Особые указания для разработчика

### Общие указания
- Реализацию нужно выполнять **поэтапно**, не пытаясь сразу собрать весь уровень без работающего core-геймплея.
- Сначала должна быть стабилизирована **базовая механика эхо**, и только потом — усложняться окружение.
- Все критические параметры должны быть вынесены в конфиг.
- Визуальный слой не должен ломать core-логику.
- Приоритет №1 — корректность игрового состояния и предсказуемость swap-механики.

### Пошаговый план реализации

#### Шаг 1. Подготовка проекта
- создать Unity-проект;
- настроить структуру папок;
- подключить и настроить Input System;
- создать bootstrap-сцену и тестовую сцену.

**Результат шага:** проект запускается, сцены открываются, структура готова к разработке.

#### Шаг 2. Реализация базового контроллера игрока
- создать игрока;
- реализовать движение;
- реализовать камеру;
- реализовать проверку состояния земли;
- добавить базовый рестарт сцены.

**Результат шага:** игрок стабильно перемещается по тестовой сцене.

#### Шаг 3. Реализация системы создания эхо
- создать `EchoManager`;
- создать префаб эхо;
- реализовать создание эхо по кнопке;
- сохранить список эх;
- реализовать лимит по количеству;
- добавить базовую визуализацию.

**Результат шага:** игрок может создавать ограниченное число статичных эх.

#### Шаг 4. Реализация выбора активного эха
- ввести понятие активного эха;
- реализовать переключение между эхо;
- добавить визуальное выделение выбранного эха;
- вывести состояние выбора в HUD.

**Результат шага:** игрок может явно выбрать, с каким эхо будет выполняться swap.

#### Шаг 5. Реализация swap-механики
- реализовать проверку валидности swap;
- реализовать обмен позициями игрока и эхо;
- добавить кулдаун;
- обработать невалидные сценарии без поломки состояния;
- добавить базовую обратную связь.

**Результат шага:** swap работает стабильно и соответствует концепции обмена, а не телепортации.

#### Шаг 6. Реализация интерактивных объектов
- создать кнопку;
- создать дверь;
- создать смертельную зону;
- создать выход;
- реализовать систему условий активации.

**Результат шага:** появляются puzzle-сценарии, завязанные на пространственную логику.

#### Шаг 7. Реализация UI и обратной связи
- вывести счётчик эх;
- вывести активное эхо;
- вывести кулдаун;
- вывести сообщения об ошибках;
- подключить базовые VFX/SFX.

**Результат шага:** игрок понимает состояние системы и причины ограничений.

#### Шаг 8. Сборка вертикального среза уровня
- собрать `Level_01`;
- расставить препятствия;
- встроить минимум одну последовательность задач, требующую нескольких эх;
- провести первичный balance pass.

**Результат шага:** уровень можно пройти только за счёт правильного использования механики.

#### Шаг 9. Тестирование и полировка
- провести ручные тесты;
- исправить критические баги;
- проверить рестарт, кулдауны, лимиты, невалидные состояния;
- подготовить краткую документацию.

**Результат шага:** прототип готов к демонстрации и дальнейшему развитию.

### Дополнительные рекомендации
- при проектировании уровней не превращать механику в “спам swap”;
- поощрять планирование и осознанное размещение эх;
- строить задачи так, чтобы игрок понимал причинно-следственную связь между размещением эха и дальнейшим решением;
- сохранять читаемость сцен: игрок должен понимать, зачем ему оставлять эхо в конкретной точке.

## Дополнение: перечень рисков, которые нужно учесть при реализации

1. **Коллизии при swap**  
   Игрок или эхо могут появляться внутри геометрии, если не выполнить корректную проверку капсулы/пространства.

2. **Потеря согласованности списка эх**  
   Возможна при удалении, рестарте, повторной инициализации уровня и неправильной работе ссылок.

3. **Сложность UX при нескольких эхо**  
   Если не сделать понятную индикацию активного эха, механика станет неудобной.

4. **Нечитаемость puzzle-сценариев**  
   Если уровень можно пройти обычным движением, уникальность механики пропадает.

5. **Нестабильный рестарт уровня**  
   Если эхо, двери и кнопки не возвращаются в исходное состояние, уровень может ломаться.
