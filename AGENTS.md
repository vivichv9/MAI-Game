# AGENTS.md

## Назначение

Этот файл задает рабочие правила для coding agents и разработчиков, которые будут создавать Unity-прототип по ТЗ `TZ_Unity_Echo_Puzzle_Game.md`.

Главная цель проекта: играбельный вертикальный срез 3D-головоломки на Unity/C#, где центральная механика строится вокруг создания эхо-позиций и обмена позициями игрока с выбранным эхо.

## Источники правды

1. `TZ_Unity_Echo_Puzzle_Game.md` - полное техническое задание.
2. `PLANS.md` - текущий план реализации, этапы, чеклисты и критерии готовности.
3. `README.md` - краткая пользовательская документация проекта, которую нужно расширять по мере реализации.

Если между документами есть противоречие, приоритет такой:

1. Новые явные указания пользователя.
2. `TZ_Unity_Echo_Puzzle_Game.md`.
3. `AGENTS.md`.
4. `PLANS.md`.
5. `README.md`.

## Контекст проекта

- Движок: Unity 2022.3 LTS или совместимая LTS-версия.
- Язык: C#.
- Платформа прототипа: Windows PC.
- Режим игры: одиночная 3D-головоломка, без мультиплеера.
- Input: Unity Input System.
- Physics: стандартная 3D-физика Unity.
- UI: выбрать один подход на старте и держаться его во всем проекте.

Стартовые решения для прототипа, если пользователь не изменит их отдельно:

- Контроллер: Third Person Controller, потому что он лучше показывает обмен игрока и эхо.
- UI: uGUI, как быстрый и привычный вариант для вертикального среза.
- Переполнение лимита эхо: запретить создание нового эхо и показать обратную связь.
- Эхо по умолчанию живут бесконечно, но архитектура должна позволять добавить `echoLifetime`.
- Критические параметры выносятся в ScriptableObject-конфиги, а не хардкодятся в gameplay-коде.

## Обязательные правила механики

- Swap - это именно обмен позициями, а не обычная телепортация.
- После swap игрок занимает позицию выбранного эхо, а выбранное эхо переносится на старую позицию игрока.
- Эхо остается активным после swap и может использоваться повторно.
- Невалидный swap отменяется полностью, без частичного переноса объектов.
- Список эхо, активный выбор и cooldown не должны ломаться при ошибках, смерти, рестарте или удалении эхо.
- Визуальный слой не должен владеть core-состоянием механики.
- Уровень `Level_01` должен требовать использования эхо, а не проходиться только обычным движением.

## Архитектурные зоны ответственности

### Core

Отвечает за запуск игры, состояние уровня, рестарт, завершение уровня и общую связку систем.

Ожидаемые классы:

- `GameBootstrap`
- `GameManager`
- `LevelManager`
- `CheckpointManager` опционально

### Player

Отвечает за ввод, движение, камеру, grounded-состояние и player-state.

Ожидаемые классы:

- `PlayerController`
- `PlayerInputHandler`
- `GroundChecker`
- `PlayerStateMachine` или простой state-controller

### Echo

Отвечает только за создание, хранение, выбор, визуальное состояние и swap эхо.

Ожидаемые классы:

- `EchoManager`
- `EchoInstance`
- `EchoSelectionController`
- `EchoSwapController` или `SwapService`
- `EchoVisual`

### Interaction

Отвечает за игровые объекты уровня: кнопки, двери, ловушки, выход, условия активации.

Ожидаемые классы:

- `InteractableBase`
- `FloorButton`
- `DoorController`
- `HazardZone`
- `ExitZone`
- `TriggerCondition` или `ConditionGroup`

### UI

Отвечает за HUD и обратную связь, но не хранит gameplay-состояние как источник правды.

Ожидаемые классы:

- `GameplayHUD`
- `EchoCounterView`
- `SwapCooldownView`
- `ActionFeedbackView`

### Data / Config

Отвечает за настраиваемые параметры.

Ожидаемые конфиги:

- `EchoConfig`
- `PlayerConfig`
- `InteractionConfig`
- `LevelGoalConfig`

## Правила C#-кода

- Используйте понятные имена классов, методов и полей.
- По умолчанию используйте `private`.
- Для инспектора используйте `[SerializeField] private`, а не лишние `public` поля.
- Публичный API должен быть минимальным и осмысленным.
- Магические числа выносите в конфиги или именованные константы.
- Не превращайте `Update()` в центр всей логики. В `Update()` допустимы ввод, движение, таймеры и короткая маршрутизация.
- Избегайте длинных монолитных методов.
- Не создавайте God Object, который управляет всем проектом.
- Нетривиальную публичную логику комментируйте кратко.
- Visual/VFX/SFX-код держите отдельно от gameplay-правил.

Если в проекте вводятся namespaces, используйте единый стиль:

```csharp
namespace MAIGame.Echo
{
}
```

Для разных областей используйте `MAIGame.Core`, `MAIGame.Player`, `MAIGame.Interaction`, `MAIGame.UI`, `MAIGame.Data`.

## Правила Unity-ассетов

- Включайте Force Text Serialization для удобного ревью сцен и префабов.
- Не удаляйте `.meta` файлы вручную.
- Не переименовывайте ассеты без необходимости, потому что это ломает ссылки в сценах и префабах.
- Для прототипа допустимы простые примитивы Unity, если они ясно читаются в gameplay.
- Префабы должны быть переиспользуемыми и не зависеть от одной конкретной сцены без причины.
- Все gameplay-настройки, которые дизайнер может менять, должны быть доступны через конфиги или serialized fields.

## Правила работы над задачей

Перед изменениями:

1. Прочитайте релевантные разделы `TZ_Unity_Echo_Puzzle_Game.md`.
2. Проверьте актуальный этап в `PLANS.md`.
3. Осмотрите существующие сцены, префабы и скрипты.
4. Делайте минимально достаточные изменения для текущего этапа.

Во время разработки:

- Держите изменения связанными с одним этапом или одной фичей.
- Не переписывайте соседние системы без необходимости.
- Если обнаружены пользовательские изменения, не откатывайте их.
- Если нужно изменить архитектурное решение из этого файла, сначала зафиксируйте причину в `PLANS.md`.
- После завершения этапа обновите чеклист в `PLANS.md`.

## Минимальные проверки перед сдачей изменений

Для кода:

- Проект компилируется в Unity без критических ошибок.
- Нет явных null-reference сценариев в основном gameplay-path.
- Основной сценарий вручную проверен в `Test_Playground` или `Level_01`.

Для механики эхо:

- Создание эхо работает в допустимых условиях.
- Лимит эхо соблюдается.
- Активное эхо выбирается и визуально выделяется.
- Swap меняет игрока и эхо местами.
- Невалидный swap не меняет состояние.
- Cooldown блокирует повторный swap.
- Рестарт очищает или восстанавливает состояние уровня согласованно.

Для уровня:

- Есть стартовая позиция.
- Есть минимум одна дверь, кнопка, опасная зона и выход.
- Уровень нельзя пройти, полностью игнорируя эхо.
- После смерти или рестарта кнопки, двери, эхо и игрок возвращаются в корректное состояние.

## Definition of Done для вертикального среза

Вертикальный срез считается готовым, когда:

- Игрок стабильно перемещается по уровню.
- Можно создать несколько эхо в пределах лимита.
- Можно выбрать активное эхо.
- Swap работает как обмен позициями.
- UI показывает количество эхо, активное эхо, cooldown и причины отказа.
- Есть полностью проходимый `Level_01`.
- Есть ручной сценарий приемки и он проходит.
- Базовые Play Mode тесты покрывают core-механику эхо.
- README объясняет запуск, управление и тестирование прототипа.
